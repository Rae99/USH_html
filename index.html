Background

The Problem

🧩
Siloed Tools
SAST and penetration testing exist as separate, disconnected solutions with no unified interface or shared job model.
🖥
Infrastructure-Heavy
Traditional scanners require long-running servers — idle compute costs money even when no scans are running.
🔒
No Unified UI
Developers have no single place to submit scans, track job status, and review severity-classified findings.
Can a fully serverless, cloud-native platform unify SAST and API Penetration Testing into a single, on-demand scanning service — without managing any servers?

Investigations (1 of 2)

What We Investigated

01
Serverless scan orchestration — ECS Fargate one-shot tasks reliably run scanner containers, report results through DynamoDB + S3, and terminate automatically with no persistent compute.
02
Unified job lifecycle API — Single REST surface where SAST and Pentest share the same job model (create → upload → start → poll → report), differentiated only by scanType.
03
Step Functions as workflow orchestrator — Handles branching, retries (2×), timeout enforcement, and conditional SNS alerting — no complex Lambda-to-Lambda chaining needed.
04
Containerizing scanner backends — Packaged provided backends into Docker images (node:18-alpine), pushed to ECR, run as ephemeral Fargate tasks injected with job-specific env vars.
05
S3 pre-signed URLs — Bypass Lambda's 6 MB payload limit; client uploads source code directly to S3, which the SAST container retrieves independently at scan time.
12
Investigations (2 of 2)

What We Investigated

06
Network isolation for scanners — Private VPC subnets + NAT Gateway; no direct internet ingress while allowing outbound ECR image pulls and pentest HTTP requests.
07
GSI-based DynamoDB schema — 3 Global Secondary Indexes (source, severity, status) enable multi-dimensional frontend filtering without full table scans.
08
Frontend polling strategy — Evaluated polling vs. WebSockets vs. SNS-to-browser; chose 10s client polling as the simplest stateless approach for low-traffic async scanning.
09
Terraform modular IaC — 8 independent modules enabled three team members to work on different AWS resources in parallel without conflicts.
10
End-to-end validation — Real scans: SAST on a test JS file (36 findings, HIGH severity) and Pentest against a vulnerable test target (6 categories, 100% detection).

Reflection

What We Learned

🔀
Step Functions are powerful but verbose — Built-in retry/error handling is excellent, but state machine YAML grows complex. AWS visual console was essential for debugging.
⏱
ECS Fargate cold starts matter — ~30s container launch is noticeable. Container warm pools or Lambda SnapStart would improve interactive experience in production.
🏫
Learner Lab constraints shape architecture — On-demand DynamoDB, IAM LabRole, and session-expiry credentials forced design decisions that differ from production best practices.
📦
Modular Terraform pays off — 8 modules let three team members work on different AWS resources in parallel without merge conflicts.
🔍
Security scanning is depth vs. breadth — Regex SAST is fast but produces false positives. AST-based or flow-sensitive analysis catches more true positives at significantly higher implementation cost.

Summary

What We Delivered

Unified scanning — SAST and Pentest under one API and React UI, scanner-agnostic by design
Zero persistent servers — Lambda + ECS Fargate on demand; one-shot containers per scan, auto-terminated
Orchestrated workflows — Step Functions: branching, 2× retries, timeouts, conditional SNS alerting
Network-isolated containers — Private VPC subnets + NAT Gateway; no direct internet ingress to scanners
Reproducible deployment — Full-stack IaC via 8 Terraform modules across all AWS resource groups

The platform is scanner-agnostic — any future backend integrates by replacing the container image.
